# 在线自适应 A/B 测试框架设计说明

> **文档摘要**：本文档从系统架构、核心模块、数据流程、API 接口、实施路径五个维度，总结"统计检验 + 自适应流量分配"在线 A/B 测试框架的设计方案。

---

## 1. 设计目标与原则

### 1.1 核心目标

将推荐算法迭代从离线实验推进到线上闭环验证，实现 **"实验配置 → 流量路由 → 数据回收 → 统计检验 → 动态调权 → 全量推优"** 的自动化链路。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **用户分桶一致性** | 同一用户在整个实验周期内始终看到同一种推荐策略，避免体验跳变 |
| **流量可配可逆** | 支持按比例划分流量，可随时调整或终止实验 |
| **指标完备可算** | 复用现有埋点表采集曝光、点击、评分、观看时长等行为，关联实验标识 |
| **统计自动决策** | Python 脚本周期性地对各策略指标进行假设检验，判断显著性与收敛性 |
| **自适应止损** | 使用 Thompson Sampling 动态调整流量分配，表现差的策略自动减少流量 |

## 2. 系统架构

### 2.1 整体架构图

```
┌─────────────────────────────────────────────────────────────┐
│                    管理层（Admin API）                       │
│  创建/修改/停止/归档实验 → 配置写入 MySQL → 内存缓存      │
└─────────────────────┬───────────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────────┐
│              流量分发中间件（Node.js Middleware）            │
│  • MD5 用户分桶（user_id → 0~99 桶号）                     │
│  • 固定分流：桶号 → 策略映射                                │
│  • 自适应分流：Thompson Sampling 采样 → user_bucket_override │
│  • 实验标识注入 req.experiment                                │
└─────────────────────┬───────────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────────┐
│                    推荐引擎                                  │
│  根据 experiment_id + strategy_id 调用对应推荐算法策略      │
└─────────────────────┬───────────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────────┐
│              用户行为埋点 → MySQL（users_movies_behaviors）  │
│        experiment_id | strategy_id 随行写入                  │
└─────────────────────┬───────────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────────┐
│         Python 离线分析模块（每 30 分钟执行）                │
│  聚合指标 → 统计检验 → Bandit 参数更新 → Redis              │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 数据流闭环

```
[实验配置] → [流量分发中间件] → [推荐引擎] → [用户行为埋点] → [MySQL]
                                                         ↓
[Python 离线分析: 聚合指标 + 统计检验 + 更新后验参数] → [Redis]
                                                         ↓
[下一次流量分发读取 Redis 后验参数，实现自适应分流]
```

## 3. 核心模块设计

### 3.1 实验配置管理（MySQL + 内存缓存）

**表结构**：

- **`ab_experiments`**：实验元信息表
  - `id, name, description, start_time, end_time, status, split_mode, created_at, updated_at`
  - `split_mode` 枚举：`fixed | bandit`
- **`ab_strategies`**：策略配置表
  - `id, experiment_id, name, algorithm_key, initial_weight, weight_source, created_at`
  - `weight_source` 枚举：`initial | bandit | manual`

**缓存策略**：服务启动时加载全部进行中实验到内存，定时任务每分钟增量更新。

### 3.2 用户分桶与流量路由中间件

**分桶算法**：

```
user_id → MD5 → 取前 8 位十六进制 → parseInt(hex, 16) % 100 → 桶号 (0~99)
```

**分流模式对比**：

| 模式 | 逻辑 | 适用场景 |
|------|------|----------|
| **固定分流 (fixed)** | 桶号 → 策略区间映射，始终不变 | 传统 A/B 测试，需要等量样本 |
| **自适应分流 (bandit)** | 新用户：从 Redis 读取 Beta(α, β) 参数 → Thompson Sampling 采样 → 记录到 `user_bucket_override` 表 | 多策略对比，快速收敛到最优 |

**中间件职责**：
1. 提取 `user_id`（未登录用设备指纹/会话ID）
2. 计算桶号
3. 遍历进行中实验，根据桶号匹配策略
4. 将 `experiment_id + strategy_id` 挂载到 `req.experiment`
5. 响应阶段将实验信息写入响应头/日志

### 3.3 埋点与数据收集

**表扩展**：在 `users_movies_behaviors` 中新增字段：
- `experiment_id` INT NULL
- `strategy_id` INT NULL

**上报策略**：
- 推荐曝光：服务端在生成推荐列表时直接写入（避免前端漏报）
- 点击/评分/收藏/播放：前端上报时，服务端自动从请求上下文获取实验标识

### 3.4 Python 离线分析模块（`ab_analysis.py`）

**执行频率**：每 30 分钟（crontab / Celery）

**核心流程**：

```
从 MySQL 读取行为数据
    ↓
按 experiment_id + strategy_id 分组
    ↓
计算 CTR、人均观看时长、评分率、收藏率 + 95% 置信区间
    ↓
统计检验：
  • 比例指标 → Z 检验 / 卡方检验
  • 均值指标 → t 检验 / Mann-Whitney U 检验
  • α=0.05, 1-β=0.8
    ↓
Bandit 模式 → 更新 Beta(α, β) 参数写入 Redis
    ↓
结果写入 ab_results 表
```

### 3.5 自适应流量决策

**Thompson Sampling**：
- 正向事件定义（CTR 优化）：点击推荐电影 或 评分 ≥ 4 分
- 采样：`random.betavariate(alpha, beta)` → 选最大值策略
- 批处理：每 10 分钟为新用户统一分配

**收敛判定**：
- 某策略"获胜概率"连续 6 个周期（3 小时）> 95%
- 且实验运行 ≥ 24 小时
- → 自动终止实验，该策略推全（流量 100%）

**冷启动保护**：
- 新策略最小流量下限 5%
- 前 2 小时不参与 Bandit 自适应调整

## 4. API 接口设计

### 4.1 实验管理 API（管理员后端）

| 方法 | 路径 | 描述 |
|------|------|------|
| POST | `/api/admin/experiments` | 创建实验（名称、策略列表、流量比例、分流模式、起止时间） |
| PUT | `/api/admin/experiments/:id` | 修改实验配置 |
| GET | `/api/admin/experiments` | 获取实验列表（支持按状态筛选） |
| GET | `/api/admin/experiments/:id` | 获取实验详情 + 策略实时指标 |
| POST | `/api/admin/experiments/:id/stop` | 手动终止实验，触发全量推优或回退 |
| POST | `/api/admin/experiments/:id/archive` | 归档实验 |

### 4.2 内部数据接口

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/internal/experiment-data/:id` | Python 获取指定实验的原始行为数据（分页） |
| POST | `/api/internal/update-bandit-params` | Python 写入 Bandit 参数到 Redis（代理接口） |
| GET | `/api/internal/bandit-params/:exp_id` | 获取 Bandit 后验参数（备用，实际可直接读 Redis） |

### 4.3 推荐服务入口

现有推荐接口（如 `GET /api/recommend/hybrid/:userId`）内部通过流量分发中间件自动注入实验策略。返回 JSON 中额外包含 `experiment_id` 和 `strategy_id` 字段，供前端埋点校验。

### 4.4 行为上报 API

现有 `POST /api/behaviors` 接口在服务端自动从请求上下文提取 `experiment_id` 和 `strategy_id`，一同写入行为表，前端无需改动。

### 4.5 指标看板 API

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/admin/experiments/:id/metrics` | 返回各策略实时指标及趋势图数据 |

## 5. 数据库设计

### 5.1 新增/修改表

```
ab_experiments          → 实验元信息
ab_strategies           → 策略配置
ab_results              → 分析结果（指标、p值、置信区间、是否显著优胜）
user_bucket_override    → 用户分桶覆盖记录（自适应模式）
users_movies_behaviors  → 新增 experiment_id + strategy_id 字段
```

### 5.2 Redis 缓存

| Key | Value | 说明 |
|-----|-------|------|
| `ab:experiment:{id}` | 实验配置对象 | 内存缓存的备源 |
| `ab:bandit:{exp_id}:{strategy_id}:alpha` | α 值 | Bandit 后验参数 |
| `ab:bandit:{exp_id}:{strategy_id}:beta` | β 值 | Bandit 后验参数 |
| `ab:experiment:stop` | 发布消息 | 通知 Node.js 更新实验状态 |

## 6. 实施建议

### 6.1 渐进式上线

```
阶段 1（验证链路） → 1% 流量内部测试实验 → 验证数据收集 + 统计脚本
阶段 2（小范围）   → 5-10% 流量 A/B 测试 → 验证分流 + 自适应算法
阶段 3（全量）     → 按需扩大 → 多实验并行
```

### 6.2 监控告警

- 实验接口延迟、分桶异常、数据丢失监控
- 策略效果异常（如 CTR 断崖下降）自动停止实验并恢复默认策略

### 6.3 数据合规

- 用户分桶标识仅用于内部实验分流
- 不泄露用户隐私
- 不改变用户基本服务体验
