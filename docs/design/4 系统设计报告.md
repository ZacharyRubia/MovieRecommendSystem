# 第四章 系统设计

## 4.1 整体功能模块设计

本系统采用前后端分离架构，后端基于 Node.js + Express 框架构建 RESTful API，前端采用 React 框架实现单页应用，数据库使用 MySQL 存储结构化数据，并引入 Qdrant 向量数据库实现基于内容的语义推荐。系统整体划分为以下核心功能模块：

### 4.1.1 用户管理模块

**功能描述**：负责用户的注册、登录、信息管理及权限控制。

- **用户注册**：支持新用户注册，使用 bcryptjs 对密码进行加密存储；第一个注册用户自动获得管理员角色，后续用户默认为普通用户。
- **用户登录**：根据用户名和加密密码进行身份验证，返回用户信息（不返回密码哈希）。
- **用户信息管理**：支持用户信息的查询、更新和删除操作。
- **角色权限控制**：基于角色字典表（roles）实现权限分级，区分管理员（admin）和普通用户（user）两种角色。

**涉及文件**：`backend/src/routes/users.js`、`backend/src/controllers/usersController.js`、`backend/src/models/User.js`

### 4.1.2 电影管理模块

**功能描述**：提供电影信息的增删改查、分类浏览、详情展示等功能。

- **电影列表**：支持分页查询，可按上映年份、评分、题材等条件过滤，结果经过缓存优化（缓存时间5分钟）。
- **电影详情**：展示电影的完整信息，包括标题、简介、封面、评分、导演、演员、标签等。
- **电影搜索**：支持按标题关键词模糊搜索。
- **题材分类**：通过电影-类型关联表实现多对多分类，支持按题材浏览电影。

**涉及文件**：`backend/src/routes/movies.js`、`backend/src/controllers/moviesController.js`、`backend/src/models/Movie.js`

### 4.1.3 用户行为与评分模块

**功能描述**：记录用户对电影的各种交互行为，为推荐算法提供数据基础。

- **行为记录**：记录用户的观看（view）、点赞（like）、收藏（collect）、评分（rate）、分享（share）、不喜欢（dislike）等行为。
- **评分功能**：用户可对电影进行1-5分评分，系统自动更新电影的平均评分（avg_rating）。
- **行为溯源**：每条行为记录包含 request_id 实现幂等性，以及客户端环境信息、来源页面等上下文数据。
- **进度追踪**：记录用户观看视频的进度（秒），支持断点续播。

**涉及文件**：`backend/src/controllers/moviesController.js`、`database/init.sql`（users_movies_behaviors 表）

### 4.1.4 评论与社交模块

**功能描述**：支持用户对电影发表评论、回复评论，以及评论的置顶管理。

- **文本评论**：用户可对电影发表文字评论。
- **嵌套回复**：支持评论的回复功能，通过 parent_id 实现评论树结构。
- **置顶管理**：管理员可将优质评论置顶（is_pinned 字段）。
- **幂等设计**：每条评论包含 request_id 保证重复提交不产生重复数据。

**涉及文件**：`backend/src/models/Comment.js`、`database/init.sql`（comments 表）

### 4.1.5 推荐系统模块

**功能描述**：系统的核心模块，综合运用多种推荐算法为用户提供个性化电影推荐。

推荐系统提供9个 API 端点：

1. **热门推荐**（`GET /api/recommend/popular`）—— 根据评分数量和评分均值计算热门电影，评分越多人评分的电影排名越高。
2. **新片推荐**（`GET /api/recommend/new-releases`）—— 按上映年份降序推荐最新电影。
3. **趋势推荐**（`GET /api/recommend/trending`）—— 基于近期评分活跃度，推荐近期被频繁评分的电影，支持7天、30天、90天时间窗口。
4. **基于内容的推荐**（`GET /api/recommend/content-based/:userId`）—— 利用 Qdrant 向量数据库，找到与用户高分电影语义相似的电影。
5. **User-Based 协同过滤**（`GET /api/recommend/user-based/:userId`）—— 寻找与目标用户评分模式最相似的邻居用户，聚合邻居用户的评分生成推荐。
6. **Item-Based 协同过滤**（`GET /api/recommend/item-based/:userId`）—— 基于用户已评分电影与其他电影的相似度，计算候选电影的预测评分。
7. **混合推荐**（`GET /api/recommend/hybrid/:userId`）—— 自适应加权融合 User-CF 和 Item-CF 的结果，权重根据用户活跃度动态调整。
8. **邻居查询**（`GET /api/recommend/neighbors/:userId`）—— 查询与用户最相似的其他用户。
9. **清除缓存**（`POST /api/recommend/clear-cache`）—— 清除推荐结果的内存缓存。

**算法核心**：

- **相似度计算**：采用 Pearson 相关系数计算用户间相似度，Cosine 相似度计算电影间相似度。
- **KNN 策略**：通过 SQL 聚合筛选共同评分超过阈值（MIN_COMMON_ITEMS=3）的候选用户，避免全量数据加载到内存。
- **稀疏数据处理**：当用户评分数据不足时（<5条），自动回退到基于题材的推荐或热门推荐。
- **缓存优化**：30分钟内存缓存存放邻居用户数据，1小时数据库缓存存放离线计算结果。
- **自适应混合权重**：根据用户评分数量动态调整 User-CF 权重：新用户（<10条）权重0.3，一般用户（10-50条）权重0.5，活跃用户（>50条）权重0.7。

**降级策略**：当 Qdrant 向量检索连接失败时，自动降级为热门推荐，保证系统可用性。

**涉及文件**：`backend/src/routes/recommend.js`、`backend/src/controllers/recommendController.js`、`backend/src/services/recommendService.js`

### 4.1.6 媒体管理与转码模块

**功能描述**：提供视频文件的存储、流式传输和转码功能。

- **媒体流服务**：支持 HTTP Range 请求，实现视频点播的拖拽播放。
- **转码功能**：支持 MKV、MP4、AVI、MOV 等常见视频格式。
- **文件管理**：支持媒体文件列表查询。

**涉及文件**：`backend/server.js`（/api/media 路由）、`backend/src/routes/transcode.js`

### 4.1.7 缓存与性能优化模块

**功能描述**：通过多级缓存策略提升系统响应速度。

- **内存缓存**：使用内存缓存存储热数据，如推荐结果、邻居用户数据，TTL 为30分钟。
- **Write-Behind 缓存**：对高并发写操作（如行为记录）采用 Write-Behind 模式，先写入缓存再异步批量刷入数据库。
- **响应缓存**：对电影列表、详情等读密集型接口，使用中间件缓存响应结果。
- **离线计算缓存**：将 ALS 等离线算法的计算结果存入数据库（movie_similarity_caches、user_recommend_caches），在线服务优先读取。

**涉及文件**：`backend/src/services/cacheService.js`、`backend/src/middleware/cacheMiddleware.js`

### 4.1.8 数据导入模块

**功能描述**：负责将 MovieLens 等外部数据源导入系统数据库。

- **电影数据导入**：导入电影基本信息、题材分类、导演、演员等。
- **评分数据导入**：导入用户评分历史数据。
- **标签数据导入**：导入电影标签和用户偏好标签。
- **Qdrant 向量导入**：将电影文本特征向量化并导入 Qdrant 向量数据库。
- **增量导入**：支持断点续传和增量数据导入。

**涉及文件**：`scripts/import_movies.py`、`scripts/import_ratings.py`、`scripts/import_tags.py`、`scripts/import_qdrant.py`

### 4.1.9 管理员模块

**功能描述**：提供系统管理功能，仅管理员角色可访问。

- **用户管理**：查看和管理所有用户。
- **电影管理**：管理电影信息的增删改。
- **系统监控**：查看缓存状态、接口调用统计。
- **数据维护**：清除缓存、触发数据同步等。

**涉及文件**：`backend/src/routes/admin.js`、`backend/src/controllers/adminController.js`

### 4.2.2 数据库 E-R 图设计

系统数据库 E-R 图展示了各实体间的关联关系。整体采用星型模式设计，以电影信息表（movies）和用户表（users）为核心实体，通过多张关联表实现多对多关系。

**核心实体关系如下**：

1. **用户（users）与角色（roles）**：多对一关系。每个用户属于一个角色，一个角色可包含多个用户。通过 users.role_id 引用 roles.id。

2. **用户（users）与电影（movies）— 行为关系**：多对多关系。通过 users_movies_behaviors 关联表实现，记录用户对电影的各种交互行为（评分、观看、收藏等）。每条行为记录包含行为类型、评分、时间戳等属性。

3. **用户（users）与标签（tags）— 偏好关系**：多对多关系。通过 users_preferred_tags 关联表实现，记录用户偏好的标签。

4. **用户（users）与电影（movies）— 评论关系**：一对多关系。一个用户可对多部电影发表评论，通过 comments 表实现，支持评论的嵌套回复（自引用 parent_id）。

5. **电影（movies）与类型（genres）**：多对多关系。一部电影可属于多种类型，一种类型可包含多部电影。通过 movies_genres 关联表实现。

6. **电影（movies）与标签（tags）**：多对多关系。通过 movies_tags 关联表实现。

7. **电影（movies）与导演（directors）**：多对多关系。通过 movies_directors 关联表实现。

8. **电影（movies）与演员（actors）**：多对多关系。通过 movies_actors 关联表实现，额外记录演员饰演的角色名（role）。

9. **电影（movies）与相似电影缓存（movies_similarities）**：一对一关系。每部电影对应一条相似电影缓存记录，存储离线计算的相似度结果。

10. **用户（users）与推荐缓存（users_recommendations）**：一对一关系。每个用户对应一条推荐缓存记录，存储离线计算的推荐结果。

**数据库设计特点总结**：

1. **标准化与反标准化结合**：核心业务数据采用第三范式（3NF）设计，确保数据一致性；同时引入 movies_similarities 和 users_recommendations 两张缓存表（JSON 字段），存储预计算结果以提升查询性能。

2. **多对多关系处理**：对电影与类型、标签、导演、演员的多对多关系，均采用独立的关联表实现，保证数据完整性和查询灵活性。

3. **行为流水设计**：users_movies_behaviors 表采用事件溯源模式，完整记录用户所有行为，为推荐算法提供丰富的数据基础。ENUM 类型限定行为范围，JSON 字段存储环境信息。

4. **幂等性保证**：users_movies_behaviors 和 comments 表均包含 request_id 唯一索引，防止重复提交导致数据重复。

5. **索引优化**：针对推荐算法的高频查询模式（按用户+行为类型+时间排序、按电影+行为类型+时间排序）建立复合索引，显著提升查询性能。

6. **级联删除保护**：所有外键均设置 `ON DELETE CASCADE`，确保删除用户或电影时，关联数据自动清理，避免孤立数据。
