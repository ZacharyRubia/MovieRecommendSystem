# 第四章 系统设计

---

## 4.1 整体功能模块设计

本系统采用前后端分离架构，后端基于 Node.js + Express 框架，前端为静态页面通过 HTTP 服务器托管，数据库使用 MySQL，推荐引擎融合协同过滤与向量检索技术。系统按功能划分为以下核心模块：

### 4.1.1 用户管理模块

用户管理模块负责用户的注册、登录、信息维护及权限控制。系统采用基于角色的访问控制（RBAC）模型，通过 `roles` 字典表定义角色（管理员和普通用户），每个用户关联一个角色 ID 以区分权限。第一个注册的用户自动晋升为管理员，后续用户默认分配普通用户角色。用户信息包括用户名、邮箱、加密后的密码哈希、头像地址等。

该模块提供的核心 API 包括：
- 用户注册（`POST /api/register`）
- 用户登录（`POST /api/login`）
- 管理员登录（`POST /api/users/admin/login`）
- 用户信息的增删改查（`/api/users`）

### 4.1.2 电影管理模块

电影管理模块是系统的核心数据模块，负责电影信息的 CRUD 操作以及电影关联信息的维护。每部电影包含标题、简介、封面图、视频资源地址、上映年份、片长、平均评分等基础信息。电影与多个实体存在多对多关联关系，包括：类型（题材）、标签、导演、演员。

管理员可以通过管理接口对电影及其关联信息进行维护，支持分页查询、标题搜索、批量导入等功能。普通用户可以通过公开接口浏览电影列表、查看电影详情。

该模块的核心 API 包括：
- 获取电影列表（`GET /api/movies`，支持分页）
- 获取电影详情（`GET /api/movies/:id`）
- 管理端电影增删改查（`/api/admin/movies`）
- 标签、导演、演员、题材的管理（`/api/admin/tags`、`/api/admin/directors`、`/api/admin/actors`、`/api/admin/genres`）

### 4.1.3 用户交互模块

用户交互模块记录和处理用户对电影的各种操作行为，包括评分、评论、观看记录以及点赞、收藏、分享等反馈行为。系统设计了统一的行为流水表 `users_movies_behaviors` 来记录所有类型的用户行为事件，每种行为通过枚举类型区分。同时支持独立的文本评论系统，评论支持多级回复和置顶功能，并通过幂等请求 ID 机制保证数据一致性。

该模块的核心 API 包括：
- 电影评分（`POST /api/movies/:movieId/rate`）
- 获取/发表/删除评论（`/api/movies/:movieId/comments`）
- 查询用户对电影的评分（`GET /api/movies/user/:userId/:movieId/rating`）

### 4.1.4 推荐系统模块

推荐系统是系统的核心功能模块，采用多种推荐算法组合的策略，为用户提供个性化的电影推荐。系统实现了以下六种推荐模式：

| 推荐模式 | 算法类型 | 说明 |
|---------|---------|------|
| User-Based CF | 协同过滤 | 基于用户的协同过滤，寻找相似用户群体，推荐邻居用户喜欢的电影 |
| Item-Based CF | 协同过滤 | 基于物品的协同过滤，根据用户历史评分推荐相似电影 |
| 混合推荐 | 协同过滤 + 自适应权重 | 融合 User-Based 和 Item-Based 结果，支持自定义或自适应权重 |
| 基于内容推荐 | 向量检索（Qdrant） | 利用 Qdrant 向量数据库进行语义相似度检索 |
| 热门推荐 | 统计排序 | 基于评分数量和质量推荐热门电影，支持按题材筛选 |
| 新片推荐 | 时间排序 | 推荐最新上线的电影 |
| 趋势推荐 | 行为热度 | 基于近期用户行为热度推荐电影（7天/30天/90天） |

系统采用写后缓存（Write-Behind Cache）策略提升性能，推荐结果支持缓存加速，并提供缓存管理 API。

该模块的核心 API 包括：
- User-Based CF（`GET /api/recommend/user-based/:userId`）
- Item-Based CF（`GET /api/recommend/item-based/:userId`）
- 混合推荐（`GET /api/recommend/hybrid/:userId`）
- 用户邻居查询（`GET /api/recommend/neighbors/:userId`）
- 热门推荐（`GET /api/recommend/popular`）
- 新片推荐（`GET /api/recommend/new-releases`）
- 趋势推荐（`GET /api/recommend/trending`）
- 基于内容推荐（`GET /api/recommend/content-based/:userId`）
- 清除推荐缓存（`POST /api/recommend/clear-cache`）

### 4.1.5 媒体服务模块

媒体服务模块负责视频文件的流式传输和管理。支持 MKV、MP4、AVI、MOV 等常见视频格式的 Range 请求分片传输，实现视频在线点播功能。同时提供媒体文件列表查询接口。

### 4.1.6 缓存服务模块

缓存服务模块为系统提供多级缓存加速能力，采用 Write-Behind 策略异步将缓存数据持久化到数据库，提升写入性能。支持对推荐结果、电影信息等高频访问数据的缓存，并提供缓存状态监控、缓存清除和数据强制刷写等管理接口。

## 4.2 数据库设计

系统采用 MySQL 关系型数据库作为主存储，存储用户、电影、行为等结构化数据。数据库名为 `MovieRecommendSystem`，字符集采用 `utf8mb4`，以支持完整的 Unicode 字符。同时使用 Qdrant 向量数据库存储电影文本特征的向量表示，用于基于内容的相似推荐。

### 4.2.2 数据库 E-R 图设计

#### 4.2.2.1 实体-关系概述

系统数据库涉及的核心实体包括：用户（User）、电影（Movie）、角色（Role）、标签（Tag）、导演（Director）、演员（Actor）、类型/题材（Genre）、评论（Comment）、用户行为（UserBehavior）。各实体之间的主要关系如下：

#### 4.2.2.2 E-R 关系描述

1. **用户与角色**：N:1 关系。一个角色可拥有多个用户，一个用户只属于一个角色。用户表通过 `role_id` 外键关联角色表。

2. **用户与电影（行为）**：N:M 关系。用户与电影之间通过 `users_movies_behaviors` 关联表记录交互行为（评分、观看、收藏等），一个用户可以对多部电影产生行为，一部电影可以被多个用户行为记录。

3. **用户与标签（偏好）**：N:M 关系。用户与标签通过 `users_preferred_tags` 关联表建立偏好关系，一个用户可选择多个偏好标签，一个标签可被多个用户偏好。

4. **电影与类型**：N:M 关系。电影与类型通过 `movies_genres` 关联表建立关系，一部电影可属于多种类型，一种类型可包含多部电影。

5. **电影与标签**：N:M 关系。电影与标签通过 `movies_tags` 关联表建立关系，一部电影可有多个标签，一个标签可标记多部电影。

6. **电影与导演**：N:M 关系。电影与导演通过 `movies_directors` 关联表建立关系，一部电影可由多位导演执导，一位导演可执导多部电影。

7. **电影与演员**：N:M 关系。电影与演员通过 `movies_actors` 关联表建立关系（附角色名属性），一部电影可有多个演员出演，一位演员可出演多部电影。

8. **电影与评论**：1:N 关系。一部电影可有多条评论，每条评论只属于一部电影。评论表通过 `movie_id` 外键关联电影表。

9. **用户与评论**：1:N 关系。一个用户可发表多条评论，每条评论只属于一个用户。评论表通过 `user_id` 外键关联用户表。

10. **评论的自引用**：1:N 关系。评论表通过 `parent_id` 自引用外键实现多级回复结构，一条评论可有多个回复。

11. **电影与相似度缓存**：1:1 关系。每部电影在 `movies_similarities` 表中有一条对应的相似度缓存记录。

12. **用户与推荐缓存**：1:1 关系。每个用户在 `users_recommendations` 表中有一条对应的推荐缓存记录。
