# 推荐算法系统设计与后续改进

## 一、推荐系统架构总览

本系统采用经典的 **"召回（Recall）→ 过滤（Filter）→ 排序（Rank）"** 漏斗架构，并结合 MySQL 关系型数据库与 Qdrant 向量数据库的双库设计，构建了一套结构严谨、可平滑升级至深度学习的工业级推荐系统。

推荐系统的完整数据流转流程如下：

1. **数据流（ETL）**：结构化数据进 MySQL，电影文本特征（标题、导演、演员、标签/题材）拼接后经 SentenceTransformers 编码存入 Qdrant 向量数据库。
2. **离线层（T+1 定时任务）**：每晚定时执行 Python 脚本，读取 MySQL 全量用户行为数据，训练 ALS 隐语义模型和 ICF 协同过滤模型，将计算结果写入 MySQL 缓存表。
3. **在线层（API 服务）**：前端请求推荐时，后端并行触发多路召回（Redis 缓存、Qdrant 向量检索、MySQL 关联查询），经去重过滤和加权重排后，组装电影元数据返回前端。

---

## 二、召回层（多路召回引擎）

召回层的目标是"快且全"，从数万部电影中快速筛选出几百部潜在感兴趣的候选集。系统采用多路召回策略，覆盖协同过滤、隐语义模型和基于内容的向量检索。

### 1. 传统协同过滤（CF）—— 基础行为召回

| 召回方式 | 算法逻辑 | 实现方式 | 优势 |
|---------|---------|---------|------|
| **Item-Based CF（ICF）** | 根据用户近期高评分电影，推荐经常被其他用户共同评分的电影（"喜欢 A 的人也喜欢 B"） | 利用 Pandas `.corr(method='pearson')` 向量化极速计算电影间相似度矩阵 | 电影属性相对稳定，推荐结果解释性强，适合放在"相似电影"模块 |
| **User-Based CF（UCF）** | 找到与当前用户口味最相似的一批"邻居"用户，推荐他们看过的电影 | Pandas 矩阵运算优化（避免原生 Python 双重循环） | 挖掘社交化推荐信号 |

**系统当前实现状态**：已实现基于用户和基于物品的协同过滤，并提供混合推荐（Hybrid）支持自定义权重或自适应权重融合。对应的 API 为：
- `GET /api/recommend/user-based/:userId`（User-Based CF）
- `GET /api/recommend/item-based/:userId`（Item-Based CF）
- `GET /api/recommend/hybrid/:userId`（混合推荐）
- `GET /api/recommend/neighbors/:userId`（邻居用户查询）

### 2. 隐语义模型（ALS 矩阵分解）—— 进阶行为召回

**核心定位**：全面替代/升级传统的 UCF 和 ICF。

**算法逻辑**：将庞大的 `(User, Movie)` 稀疏评分矩阵分解为"用户隐特征向量"和"电影隐特征向量"，预测评分即为两个向量的点积。

**实现优势**：
- 使用 `implicit` 库结合 `scipy.sparse`，在内存友好的情况下利用 CPU 多核极速完成百万级数据训练
- 生成的 User 向量和 Item 向量不仅能计算评分预测，还能直接计算出更高质量的用户相似度和物品相似度
- ALS 的离线计算结果可写入 MySQL 缓存表（`users_recommendations`、`movies_similarities`），线上通过 Redis 加速读取

**系统当前实现状态**：已完成 ALS 模型的离线训练脚本（`scripts/recommend/train_recommend.py`），实现了模型文件的持久化保存（`/models/` 目录），并集成了离线预测写入缓存表的能力。

### 3. 基于内容的向量召回（Content-Based）—— 解决冷启动

**算法逻辑**：利用大语言模型（SentenceTransformers）将电影的文本特征（标题 + 导演 + 演员 + 标签/题材）拼接后编码为高维向量，存入 Qdrant 向量数据库。推荐时，将用户画像或历史行为向量化后在 Qdrant 中进行近似最近邻（ANN）搜索。

**关键应用场景**：
- **新电影冷启动**：新上线的电影无人评分，CF 无法计算，但可通过 Qdrant 搜索与之题材相似的旧电影，推荐给喜欢该类题材的用户
- **新用户冷启动**：新用户勾选偏好标签（写入 `user_preferred_tags` 表），转换为文本向量后去 Qdrant 召回初始候选电影

**系统当前实现状态**：已集成 Qdrant 向量数据库，实现了电影文本特征的向量化导入脚本（`scripts/import_qdrant.py`），并提供了基于内容的推荐 API（`GET /api/recommend/content-based/:userId`）。

## 三、过滤层（Filter）

从多路召回引擎获取候选电影后，必须进行业务规则过滤：

1. **去重**：合并多路召回的结果，消除重复电影
2. **历史行为过滤**：查询 `users_movies_behaviors` 表，剔除用户已看过的、点过"不喜欢"（dislike）的电影
3. **黑名单/下架过滤**：剔除已下架或不可用的视频资源

**系统当前实现状态**：过滤逻辑在推荐服务层（`recommendService`）中实现，支持缓存加速和超时保护机制。

## 四、排序与返回层（Rank & Return）

候选集被过滤后，需要精确打分以决定最终展示给用户的 Top N 顺序。

### 1. 当前实现：规则打分与加权融合（Hybrid 策略）

**算法逻辑**：将多路召回（UCF、ICF、ALS、Qdrant 内容召回）的分数归一化后加权相加，得到最终评分。例如：
- ALS 预测分占 40%
- ICF/协同过滤占 30%
- Qdrant 向量相似度占 30%

**微调因子**：可结合 MySQL `movies.avg_rating`（电影全局平均分）和 `release_year`（上映年份），给高分经典或最新大片一定的提权。

**系统当前实现状态**：推荐控制器（`recommendController.js`）实现了以下 9 个 API 端点：

| 端点 | 功能 | 说明 |
|------|------|------|
| `GET /api/recommend/user-based/:userId` | User-Based CF | 基于用户的协同过滤推荐 |
| `GET /api/recommend/item-based/:userId` | Item-Based CF | 基于物品的协同过滤推荐 |
| `GET /api/recommend/hybrid/:userId` | 混合推荐 | 融合 UCF + ICF，支持自适应权重 |
| `GET /api/recommend/neighbors/:userId` | 邻居查询 | 查找最相似用户 |
| `GET /api/recommend/popular` | 热门推荐 | 基于评分数量和质量的排序 |
| `GET /api/recommend/new-releases` | 新片推荐 | 按上映时间排序 |
| `GET /api/recommend/trending` | 趋势推荐 | 基于近期行为热度（7/30/90 天） |
| `GET /api/recommend/content-based/:userId` | 内容推荐 | Qdrant 向量检索 |
| `POST /api/recommend/clear-cache` | 清除缓存 | 清空推荐缓存 |

### 2. 未来规划：深度学习排序模型

在现有基础架构完善后，可通过深度学习技术彻底升级召回和排序能力：

#### 双塔模型（Two-Tower Model）—— 用于高效召回

**架构设计**：
- **左塔（用户侧）**：输入用户特征（用户画像、历史观看序列、行为序列），经多层神经网络（DNN）编码为用户 Embedding 向量
- **右塔（物品侧）**：输入物品特征（电影元数据、标签、题材），经多层 DNN 编码为物品 Embedding 向量
- **线上推理**：将所有电影的右塔向量离线刷入 **Qdrant**。用户请求时，实时计算左塔向量后，在 Qdrant 中进行高效的 ANN（近似最近邻）搜索

**结合现有系统**：右塔向量可直接复用以 Qdrant 为基础的向量检索基础设施，实现无缝升级。

#### NCF（Neural Collaborative Filtering）—— 用于精准排序

**架构设计**：
- 传统矩阵分解将 User 和 Item 向量做简单内积（线性乘法）
- NCF 将两个向量拼接后输入多层感知机（MLP），让神经网络学习高阶非线性交叉特征
- 输出最终的点击率/评分预测

**适用场景**：对召回后的小规模候选集（约 100 部电影）进行精确排序，决定最终展示给用户的 Top 20。

## 五、系统当前实现总结

| 阶段 | 模块 | 当前状态 | 实现方式 |
|------|------|---------|---------|
| 召回 | User-Based CF | ✅ 已实现 | Pandas 矩阵运算 + Pearson 相关系数 |
| 召回 | Item-Based CF | ✅ 已实现 | Pandas 相似度矩阵计算 |
| 召回 | ALS 隐语义模型 | ✅ 已实现 | `implicit` + `scipy.sparse`，离线训练与预测 |
| 召回 | 基于内容召回（Qdrant） | ✅ 已实现 | SentenceTransformers + Qdrant 向量检索 |
| 召回 | 双塔模型 | 🔲 规划中 | 需神经网络架构设计与训练流程 |
| 过滤 | 去重/去已看/黑名单 | ✅ 已实现 | 基于 MySQL 行为表过滤 |
| 排序 | 加权融合排序 | ✅ 已实现 | Hybrid 自适应权重策略 |
| 排序 | NCF 深度排序 | 🔲 规划中 | 需 MLP 模型训练与部署 |
| 缓存 | 推荐结果缓存 | ✅ 已实现 | MySQL 缓存表 + 缓存管理 API |

## 六、后续改进方向

### 近期改进（基于现有架构优化）

1. **推荐质量评估**：引入离线评估指标（精确率、召回率、NDCG、覆盖率、新颖度），基于 MLflow 等工具有效对比不同算法的推荐效果
2. **冷启动策略完善**：结合 `user_preferred_tags` 表的新用户偏好注册流程，进一步优化新用户的初始推荐体验
3. **实时性提升**：将 MySQL 缓存表迁移至 Redis，降低推荐延迟，支持秒级缓存更新
4. **AB 测试框架**：建设在线 AB 测试平台，对比不同推荐算法组合的线上实际效果

### 中期改进（深度学习引入）

1. **双塔模型召回**：利用用户和物品的 DNN 编码替代手工特征工程，在大规模候选集中实现高效召回
2. **NCF 精准排序**：引入多层神经网络对候选集进行精细排序，提升推荐列表的点击率和满意度
3. **特征体系构建**：建设统一的特征工程管道，整合用户行为特征、物品属性特征、上下文特征等

### 长期改进（智能化与效率提升）

1. **多任务学习**：在深度模型中同时优化多个目标（点击率、观看时长、评分、收藏行为等）
2. **强化学习**：引入强化学习实现推荐策略的动态调整，根据用户实时反馈优化推荐序列
3. **自动化 ML**：构建 AutoML 管道，自动搜索最优模型结构和超参数配置
4. **联邦学习**：在保护用户隐私的前提下，实现跨用户群体的协同模型训练
