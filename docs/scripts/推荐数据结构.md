根据对项目代码的完整分析，以下是两张缓存表中存储数据的具体含义：

---

## 1. `users_recommendations` — 用户推荐缓存表

**表结构**（定义在 `database/init.sql` 第 228-234 行）：
```sql
CREATE TABLE `users_recommendations` (
    `user_id`     BIGINT UNSIGNED PRIMARY KEY,   -- 用户 ID
    `recommend_movies` JSON NOT NULL,             -- 推荐电影列表(JSON)
    `algorithm`   VARCHAR(20) NOT NULL,           -- 算法类型
    `updated_at`  TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

**存储的数据含义**：
| 列名 | 含义 |
|------|------|
| `user_id` | 目标用户 ID |
| `recommend_movies` | **推荐给该用户的电影列表**（JSON 数组），每项包含 `movie_id`（电影 ID）和 `score`（预测评分/推荐置信度） |
| `algorithm` | 生成推荐的算法标识，取值如 `svd`、`user_cf`、`item_cf`、`hybrid`、`als` 等 |
| `updated_at` | 该缓存记录的最后更新时间 |

**写入来源**（来自 `save_to_cache.py` 第 72-127 行 和 `recommendService.js` 第 26-43 行）：
- Python 端：`save_to_cache.save_user_recommendation(user_id, recommendations, algorithm)` 通过 `REPLACE INTO users_recommendations` 写入
- Node.js 端：`saveCacheUserRecommend(userId, recommendations, algorithm)` 在实时计算完成后异步写回

**数据用途**（来自 `recommendService.js` 第 215-232 行）：
- **缓存优先**：后端在收到推荐请求时，**先查询此表**（`getCachedUserRecommend`），若缓存未过期（TTL=1 小时）则直接返回，避免重复计算
- 这是典型的 **Cache-Aside 模式**——离线/实时计算结果先写入此表，线上请求优先命中缓存

**你展示的数据解读**（第一段）：
```json
[{"score": 0.9967, "movie_id": 11}, {"score": 0.0168, "movie_id": 10}, {"score": -0.0248, "movie_id": 2}]
```
- 用户 1/2/3 的推荐结果完全相同，`algorithm='als'` 说明可能是 **ALS（交替最小二乘法）** 模型计算的结果
- 推荐了 **3 部电影**：电影 11（评分 0.9967，极高置信度）、电影 10（评分 0.0168）、电影 2（评分 -0.0248，负分）
- 注意三条记录的 `algorithm` 都是 `als` 而非系统中常见的 `svd`/`hybrid`——这表明系统中还存在一个**基于 Spark ALS 的推荐管道**，与这组 Python 训练的模型是并行的

## 2. `movies_similarities` — 电影相似度缓存表

**表结构**（定义在 `database/init.sql` 第 215-220 行）：
```sql
CREATE TABLE `movies_similarities` (
    `movie_id`      BIGINT UNSIGNED PRIMARY KEY,  -- 电影 ID
    `similar_movies` JSON NOT NULL,                -- 相似电影列表(JSON)
    `updated_at`    TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

**存储的数据含义**：
| 列名 | 含义 |
|------|------|
| `movie_id` | **源电影**（要找与谁相似的电影） |
| `similar_movies` | **与源电影相似的电影列表**（JSON 数组），每项包含 `movie_id`（相似电影 ID）和 `similarity`（相似度分数 0~1） |
| `updated_at` | 该缓存记录的最后更新时间 |

**写入来源**（来自 `save_to_cache.py` 第 134-188 行 和 `recommendService.js` 第 48-65 行）：
- Python 端：`save_to_cache.save_movie_similarity(movie_id, similar_movies)` 通过 `REPLACE INTO movies_similarities` 写入
- Node.js 端：`saveCacheMovieSimilarity(movieId, similarMovies)` 在 Item-Based CF 实时计算完成后异步写回

**数据用途**（来自 `recommendService.js` 第 141-210 行）：
- **Item-Based CF 的核心加速**：`getItemBasedFromCache` 方法**批量查询**用户已评分电影的相似缓存，按评分加权聚合快速生成推荐
- **减少实时计算量**：如果没有此缓存，Item-Based CF 需要实时用 SQL 计算每对电影的 Cosine 相似度（见 `recommendService.js` 第 693-846 行），耗时非常大

**你展示的数据解读**（第二段）：
```json
[{"movie_id": 10, "similarity": 1.0}, {"movie_id": 2, "similarity": 1.0}, {"movie_id": 62, "similarity": 1.0}]
```
- 电影 2、10、11 的相似电影列表完全相同，且 similarity=1.0 —— 这很可能是**测试阶段的占位数据**（Item-Based CF 的训练产物），正常情况 similarity 应该是 0~1 之间的小数
- 电影 17 的相似电影 similarity=0.7809，相对更合理
- 电影 25 对应的是另一组电影（3078、3030、2997），说明这些电影属于不同的"相似簇"

## 总结对比

| 特性 | `users_recommendations` | `movies_similarities` |
|------|------------------------|----------------------|
| **主键** | `user_id` | `movie_id` |
| **存储内容** | 给用户的推荐电影 + 预测评分 | 与源电影相似的电影 + 相似度 |
| **算法来源** | SVD / User-CF / Item-CF / Hybrid / ALS | Item-Based CF（Cosine 相似度） |
| **被谁消费** | 前端推荐页 → 后端 `getCachedUserRecommend` | Item-Based CF 推荐 → `getItemBasedFromCache` |
| **缓存策略** | **Cache-Aside**（先查缓存，未命中则实时计算并写回） | **实时计算回写**（计算完成后缓存供下次加速） |
| **数据时效** | TTL=1 小时，过期后重新计算 | TTL=1 小时，过期后重新计算 |
| **写回时机** | 实时推荐计算完成后异步写回 | 实时 Item-CF 计算完成后异步写回 |

两张表共同构成了系统的**两级缓存架构**：
1. **离线层**（离线脚本计算 → 批量导入这两张表）
2. **实时层**（在线实时计算 → 异步写回这两张表）
3. **查询层**（API 优先读取缓存 → 命中直接返回 → 未命中触发实时计算）
