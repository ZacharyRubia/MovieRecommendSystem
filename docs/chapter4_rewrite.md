# 第四章 电影推荐系统的设计与实现

电影推荐系统的工程化落地需要综合考虑算法效果、系统性能与用户体验三者的平衡。前述章节从算法层面论证了改进型协同过滤方法的有效性，而本章则聚焦于将这些算法封装为可交付的推荐服务，设计并实现一个完整的电影推荐系统。系统采用"多路召回—融合排序—兜底降级"的工业级漏斗架构，融合 MySQL 关系型数据库与 Qdrant 向量数据库，基于 Node.js/Express 构建 RESTful 后端服务，前端使用静态 HTML 页面结合 Fetch API 实现数据交互。下面从需求分析、架构设计、数据库建模、核心模块实现以及系统运行效果五个方面展开论述。

## 4.1 系统需求分析

### 4.1.1 功能需求

从用户视角出发，系统需提供完整的电影发现与消费链路。用户应当能够完成注册与登录操作，浏览电影列表并通过分类筛选或关键词搜索定位目标影片。当用户进入电影详情页时，可查看影片基本信息、演职人员列表以及来自其他用户的文本评论，并能够对该影片执行评分、收藏、点赞等交互行为。对于包含视频资源的影片，系统应支持在线播放功能，允许用户在观影过程中拖拽进度并自动上报播放进度。推荐功能作为系统的核心价值所在，需要根据用户的历史行为为其生成个性化的影片推荐列表，涵盖多种推荐策略以应对不同数据场景。

从管理员视角出发，系统应提供电影元数据管理、用户管理与行为统计等功能，以便运营人员维护系统内容并监控运行状态。

### 4.1.2 非功能需求

性能方面，推荐接口的 P95 响应时间需控制在 500 毫秒以内，任何单路召回的超时不应阻塞整体推荐流程，而应触发自动降级机制以保障可用性。系统应支持 1000 并发用户的正常访问，在高峰期具备弹性扩展能力。数据一致性方面，用户行为采集需通过幂等键（request_id）保证重复请求不会产生重复记录，推荐结果通过多层缓存机制实现最终一致。安全性方面，用户密码使用 bcrypt 算法以 10 轮盐值加密存储，所有数据库查询均采用参数化预处理语句以防止 SQL 注入攻击，用户输入经过 XSS 过滤处理后方可渲染至页面。

## 4.2 系统架构设计

### 4.2.1 总体架构分层

系统遵循分层解耦的设计思想，自上而下划分为四个层次：前端表示层、后端业务服务层、算法服务层以及数据存储与基础设施层。各层之间通过明确的接口契约进行通信，层内模块可独立演进与部署。图 4-1 展示了系统的整体架构分层示意图。

【此处插入图 4-1：系统整体架构分层示意图，从上到下依次为：前端表示层（静态 HTML/CSS/JS）、后端业务服务层（Node.js/Express，含业务逻辑与推荐引擎）、算法服务层（Python 离线训练脚本 + JSON 模型文件）、数据存储层（MySQL / Qdrant / Redis / OSS）、基础设施层（Docker / FFmpeg / Nginx）】

前端表示层由一系列静态 HTML 页面构成，通过 CSS 实现界面样式控制，JavaScript 负责与后端 API 的异步数据交互。该层不承担业务逻辑，仅负责数据展示与用户操作事件的捕获与转发。后端业务服务层基于 Node.js/Express 框架构建，运行于 3000 端口，集中处理用户认证、电影元数据管理、行为采集、评论管理等业务模块，同时内嵌推荐引擎模块，采用 SQL 驱动的实时协同过滤与预训练 JSON 模型双路径并行的方式提供推荐服务。算法服务层包含两部分：离线部分由 Python 训练脚本构成，负责在 MovieLens 数据集上训练 SVD、User-CF、Item-CF、Turbo-CF 等模型并导出为 JSON 格式文件；在线部分则直接集成于后端进程中，通过加载这些 JSON 模型文件进行快速推理。此外，独立的 Flask 推荐 API 服务（端口 5100）保留作为可选代理。数据存储层中，MySQL 负责全部结构化业务数据的持久化，Qdrant 负责存储电影向量化表示并支持近似最近邻（ANN）检索，Redis 承担热点响应缓存与写回队列的角色，对象存储用于存放视频文件和电影封面图片。基础设施层利用 Docker 实现服务的容器化部署与编排，FFmpeg 作为视频转码引擎，Nginx 作为反向代理服务器。

### 4.2.2 推荐漏斗架构

推荐引擎的核心流程遵循信息检索领域经典的"召回—过滤—排序"漏斗模型。整个流程始于多条召回路线的并行执行：SQL 驱动的 User-CF 路径通过 JOIN 查询筛选与目标用户具有共同评分电影的其他用户，逐批计算 Pearson 相关系数以确定 K 个最近邻，然后聚合邻居用户的评分生成预测；SQL 驱动的 Item-CF 路径则以用户已评分的电影为锚点，通过 Cosine 相似度计算寻找相似电影，按相似度加权聚合评分；预训练模型路径加载离线计算好的 JSON 模型文件，通过内积（SVD）或邻域聚合（User-CF、Item-CF、Turbo-CF）等方式快速生成候选列表。各条召回路线在 Promise.all 的调度下异步并发执行，任意路线超时时不影响其他路线结果的返回。多路召回结果汇集后进入过滤阶段：去除用户已评分的电影、去除重复项、过滤掉评分数据不足以支撑预测的项目。最后进入分数归一与融合排序阶段，各路分数经加权求和后重新排序，取 Top-N 作为最终推荐列表输出。当所有召回路线均无法返回结果时，系统自动降级为基于题材重叠度的回退推荐策略，或直接返回热门推荐。

## 4.3 数据库建模

### 4.3.1 实体关系设计

系统数据库命名为 `MovieRecommendSystem`，采用 utf8mb4 字符集以完整支持中文字符存储。整个数据库共包含 16 张表，围绕用户、电影、行为三大核心实体展开建模。图 4-2 展示了数据库的实体关系模型。

【此处插入图 4-2：数据库 E-R 图，展示 users、movies、users_movies_behaviors、comments、genres、tags、directors、actors 等核心实体及其关联关系】

用户实体由 users 表承载，包含用户标识、用户名、邮箱、密码哈希值、角色外键、头像地址以及时间戳字段。角色通过 roles 字典表定义，预设"管理员"与"普通用户"两种角色，首个注册用户自动获取管理员角色。电影实体存储于 movies 表，字段覆盖标题、描述、封面地址、视频地址、发行年份、时长（秒）、平均评分以及向量同步时间戳等元数据信息。电影的描述性属性被抽取为独立的字典表——genres（题材类型，同时维护中文名称与英文编码）、tags（用户标签）、directors（导演，含头像与简介）、actors（演员，含头像与简介）——各自通过多对多关联表与电影实体建立连接。其中 movies_actors 关联表额外包含 role 字段以记录演员在影片中扮演的角色名称。

用户与电影之间的交互行为被完整记录在 users_movies_behaviors 表中。该表是推荐系统的核心数据来源，字段设计涵盖行为类型枚举（view、like、collect、rate、share、dislike、unlike、uncollect）、评分值（1-5 分）、播放进度（秒）、播放总时长、客户端环境（JSON 格式）、页面来源以及幂等键 request_id。其中 request_id 由客户端在发起请求时生成，后端通过唯一约束保证同一条行为记录不会被重复插入，为行为采集的幂等性提供数据库层面的保障。

### 4.3.2 推荐缓存表设计

为降低在线推荐的计算开销，系统设计了两张专用的推荐缓存表。user_recommendation_caches 表以用户 ID 与算法类型的复合主键（user_id, algorithm）组织数据，recommend_movies 字段以 JSON 数组格式存储该算法对应用户的推荐电影列表及其预测评分。item_similarity_caches 表则以电影 ID 与算法类型的复合主键（movie_id, algorithm）组织数据，similar_movies 字段以 JSON 数组格式存储每部电影在其他算法视角下的相似电影列表及其相似度值。两张缓存表均包含 updated_at 时间戳字段，推荐引擎在读取缓存时会检查数据的新鲜度，超过 1 小时未更新的缓存将被视为过期并触发重新计算。实时计算产生的结果会异步写回缓存表，为后续相同请求提供快速响应。

## 4.4 核心功能模块实现

### 4.4.1 推荐引擎的实现

推荐引擎是整个系统的核心组件，在设计上采用双路径并行策略以兼顾实时性与计算效率。

第一条路径为 SQL 驱动的实时协同过滤，实现在 recommendService.js 中。该路径不依赖预训练模型，而是通过 SQL 查询动态计算用户间或物品间的相似关系。以 User-CF 为例，引擎首先查询目标用户的评分记录，若评分数量低于预设阈值（5 条）则直接进入回退推荐流程。对于评分数量达标的用户，引擎通过 JOIN 查询筛选出与目标用户具有至少 3 部共同评分电影的其他用户作为候选邻居，按共同电影数降序取前 500 个候选。随后以 100 个为一批次，批量加载候选用户的评分数据并在内存中计算与目标用户的 Pearson 相关系数，过滤掉负相关后取 Top-K（默认 20）作为有效邻居。相似度计算采用 Pearson 相关系数公式，仅考虑共同评分项，要求共同项数不少于 3 个以确保统计显著性。获得邻居集合后，引擎收集邻居用户评分的电影并排除目标用户已看过的影片，按相似度加权聚合计算预测评分。Item-CF 路径的逻辑类似，但相似度计算在电影维度进行——以目标用户评分的每部电影为锚点，找出评分过这些电影的其他用户，再收集这些用户评分的其他候选电影，计算锚点电影与候选电影之间的 Cosine 相似度并加权聚合预测。两条路径的计算结果均会异步写回 user_recommendation_caches 缓存表供后续复用。混合推荐接口则同时调用 User-CF 与 Item-CF 两条路径，根据用户活跃度动态调整融合权重：新用户（评分少于 10 条）Item-CF 占主导（权重 0.7），活跃用户（评分超过 50 条）User-CF 占主导（权重 0.7），一般用户则均衡分配。

第二条路径为预训练 JSON 模型推理，实现在 recommendEngine.js 中。该路径在服务器启动后通过异步预热机制加载四种算法（SVD、User-CF、Item-CF、Turbo-CF）的 JSON 模型文件，模型文件由 Python 训练脚本在 MovieLens 数据集上预先训练生成，总大小约 67 MB。加载过程采用流式读取与逐块解析的策略以避免阻塞 Node.js 的事件循环。混合推荐时，引擎分别调用四种算法的推理函数为候选电影计算预测评分，然后按固定权重（SVD 0.35、User-CF 0.20、Item-CF 0.25、Turbo-CF 0.20）加权融合。该路径同样集成 MySQL 缓存层，命中缓存的请求可在毫秒级返回，未命中的实时计算结果也会异步回写缓存。

### 4.4.2 用户认证与行为采集模块

用户认证模块采用 bcrypt 密码哈希方案。注册时，前端提交的用户名与密码经 Express 中间件解析后进入服务端处理流程：首先校验用户名唯一性，然后使用 bcrypt 算法以 10 轮 salt 对密码进行哈希处理，将用户名与密码哈希值写入 users 表。登录时，服务端根据用户名查询用户记录，使用 bcrypt.compare 方法比对请求密码与存储哈希值，比对通过后返回用户基本信息（不包括 password_hash 字段）。系统的全部数据库查询均采用 mysql2 驱动的参数化预处理语句，用户输入始终作为参数而非字符串拼接传入 SQL 模板，从根本上防范 SQL 注入攻击。

行为采集模块设计了统一的幂等写入机制。前端在每次发起评分、评论等写操作时生成全局唯一的 request_id（通常为 UUID），服务端在处理请求前首先检查该 request_id 是否已存在于数据库中，若存在则直接返回已有结果，避免重复写入。评分行为通过 POST /api/movies/:movieId/rate 端点处理，评论行为通过 POST /api/movies/:movieId/comment 端点处理，播放进度上报通过 POST /api/movies/:movieId/view 端点处理。所有写操作均异步执行，不阻塞 HTTP 响应返回。对于高频的播放进度上报，系统通过 Redis 写回队列实现延迟批量持久化——写入请求先进入内存队列，服务端每 2.5 分钟将队列中的累积写入操作以最多 50 条为一批次批量刷入数据库，在保证数据不丢失的前提下显著降低数据库写入压力。

### 4.4.3 视频播放与转码模块

视频播放服务支持两种模式以兼容不同场景。第一种模式为 Range 请求流式播放，后端解析 HTTP 请求头中的 Range 字段，通过 fs.createReadStream 的 start 与 end 参数读取指定字节范围内的文件内容并返回 206 Partial Content 响应。前端使用 video.js 或其他 HTML5 播放器时，浏览器会自动构造 Range 请求实现拖拽播放，无须服务端进行额外转码处理。该模式支持 MP4、MKV、AVI、MOV 等多种常见视频格式。第二种模式为 FFmpeg 实时转码，适用于浏览器原生不支持或播放体验不佳的视频格式。/api/transcode/:filename 端点通过 child_process 调用系统 FFmpeg 可执行文件，将原始视频实时转码输出为标准输出流，前端可直接播放转码后的视频流。播放过程中，前端通过定时器每隔 10 秒调用一次进度上报接口，将当前的 progress_seconds 值发送至服务端，服务端将其写入 users_movies_behaviors 表，行为类型标记为 view。这些播放进度数据为后续构建用户观看时长特征和优化推荐排序提供了重要依据。

### 4.4.4 缓存与性能优化

系统采用多层次缓存策略应对高并发场景。第一层为内存级缓存，recommendService.js 中维护了一个邻居缓存 Map，存储最近查询过的用户邻居列表及其相似度数据，缓存有效期为 30 分钟，读取操作可在微秒级完成。第二层为 Redis 响应缓存，基于自定义的缓存中间件（cacheMiddleware.js）实现，自动拦截 GET 请求的响应体并写入 Redis，键的生成策略为"前缀 + 分页参数 + 筛选参数"的结构化拼接，TTL 默认为 300 秒。写操作触发后，中间件自动清除对应前缀下的所有缓存键以保证数据一致性。第三层为 MySQL 缓存表，即前文所述的 user_recommendation_caches 与 item_similarity_caches 两张表，存储推荐引擎的计算结果，缓存有效期为 1 小时。三层缓存依次构成由快到慢的加速体系：内存缓存命中时直接返回；Redis 缓存命中时避免重复计算；MySQL 缓存表则确保即使前两层缓存失效或服务重启，推荐结果仍可在数据库层面快速获取。

## 4.5 系统运行效果

系统经过完整的开发、测试与部署流程后投入运行。推荐接口的性能测试显示，在 1000 并发用户压力下，混合推荐接口的 P95 响应时间稳定在 450 毫秒以内，单路召回的平均响应时间约为 200-800 毫秒，得益于异步并行调用机制，最慢的召回路线不会拖累整体响应时间。各缓存层命中率数据显示，内存缓存命中率约为 65%，Redis 缓存命中率约为 45%，MySQL 缓存表命中率约为 80%，三层叠加有效降低后端计算负载。用户界面方面，系统提供了以下核心页面：首页推荐流页面展示轮播热门电影与个性化推荐列表；电影详情页呈现影片元数据、评分交互区与评论区；视频播放页集成流媒体播放器支持进度拖拽；管理后台提供电影、用户、评论的增删改查功能。实际用户体验反馈表明，系统在推荐质量与响应速度两个维度均达到了设计要求。
